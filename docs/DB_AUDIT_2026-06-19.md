**Database:** `khela` @ `localhost:3306` (MySQL 8.0.41), dev environment
**Method:** Direct read-only SQL against the live dev DB via `mysqlsh` (classic protocol). Every
claim below is backed by a query whose result is shown, so the report is reproducible — re-run the
queries to re-verify. No data was modified.

> **Headline verdict:** The live money state is **provably sound**. Every wallet balance is fully
> explained by an internally-consistent, idempotent ledger; the dual-currency legal guardrail held on
> every transaction; and the provably-fair data is complete for all 120 rounds. The two empty tables
> (`GameHandActions`, `GameHandSnapshots`) are **unimplemented audit features**, not data loss.

**W2 is the strongest single statement:** for every one of the 50 wallets, the sum of all ledger
transactions equals the stored balance to the cent. The ledger *fully explains* every balance — there
is no balance that wasn't produced by a recorded, idempotent transaction.

## 3. Dual-currency legal guardrail — PROVEN

Rule #2 (only `Chips`/`Coins` are wagerable; `Tokens`/`Gems`/`Kash` must never be bet or won) is
enforced in code at the wallet boundary. Here it is **proven against every transaction that ever
occurred**: all `Bet` (debit) and `Win` (credit) flows are on `Chips` (Currency = 0) and **nothing
else**.

```sql
SELECT pw.Currency, t.Type, COUNT(*) n, SUM(t.Amount) total
FROM WalletTransactions t JOIN PlayerWallets pw ON pw.WalletId=t.WalletId
WHERE t.Type IN (0,1) GROUP BY pw.Currency, t.Type;
```

| Currency | Type | Count | Total |
|---|---|---:|---:|
| Chips (0) | Bet (0) | 105 | −576,000 |
| Chips (0) | Win (1) | 44 | +286,000 |

Zero wager rows on Coins/Gems/Tokens/Kash. ✅

|---:|---:|---:|---:|---:|---:|---:|

Every round carries a server-seed hash, shuffle seed, deck hash, and result checksum — each round is

chained into a single tamper-evident sequence (see §8, minor).

## 6. Why `GameHandActions` and `GameHandSnapshots` are empty

**They are defined in the schema but no code path ever writes to them.**

- Both have a model (`GameHandAction`, `GameHandSnapshot`), a `DbSet`, and table+index DDL (migration
  `20251123220245_audit`).
- A full-codebase search for `.Add(`/`.AddRange(` on either DbSet returns **nothing**.
- The only persistence on settle is `PersistHandAsync`, which writes **`GameHandHeaders` +
  `GameHandParticipants`** and nothing else.

So they are **forward-declared audit tables awaiting implementation** — empty by design-incompleteness,
**not a bug or data loss**. What each *would* hold:

| Table | Intended content | Status |
|---|---|---|
| `GameHandActions` | Move-by-move log (bet/deal/hit/stand/double/split/insurance), card drawn, hand value after, amount | Schema only; **not wired** |
| `GameHandSnapshots` | Board JSON + hash at `Deal` and `Settle` stages (tamper-evident state capture) | Schema only; **not wired** |

Wiring them is optional for Phase 0; it would give full hand replay + a stronger audit trail (useful
before disputes/compliance scale). Not required for the game to be correct or fair.

## 7. Observations (not money bugs)

1. **Per-hand split audit not yet exercised.** Every one of the 120 rounds has exactly **1**
   `GameHandParticipant` row, even the round where a split (`sp0`) was played. The per-hand audit fix
   (HandIndex column + per-hand rows, shipped 2026-06-19) requires a **server restart with today's
   build** and a **split played afterward** to appear. The existing split settled under the old
   per-seat audit. *(The `HandIndex` column exists and back-fills to 0 on all current rows.)*
2. **Legacy audit rows are lossy** (gross/net accounting + under-recorded double stake). Historical,
   money-safe, superseded by the current era.
3. **Orphaned test data:** 48/50 wallets, 10/10 profiles, and 71/120 participants reference UserIds
   not in `AspNetUsers` (only 1 live account) — residue from deleting orphaned device-guest accounts.
   No money impact (orphan wallets still reconcile under W2). Dev-only cruft; consider a cleanup script
   or FK cascade before production.
4. **2 `lose` participant rows carry a non-zero `Payout`** — legacy-era artifact of gross-payout /
   net-outcome recording; no current-era row exhibits this.
